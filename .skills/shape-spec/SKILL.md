# Shape Spec

将原始想法转化为可构建的规格文档。

## 工作流

### Phase 1: 理解
- 接受：原始想法、issue、或观察
- 探索相关代码区域（文件、模式、约束）
- 搜索现有代码库中的类似实现

### Phase 2: 产品探索
**锁定产品方向前不写代码。**

1. 调查：问题空间、用户影响、现有方案
2. 头脑风暴：2-3 个方案，分析权衡，**推荐一个**
3. 讨论：一个问题一个问题迭代，直到锁定
4. 草拟规格：目标、非目标、验收标准

### Phase 3: 技术探索
1. 探索：3-5 个技术方案，分析架构、文件变更、模式、工作量
2. **推荐一个**
3. 讨论：设计没准备好前不实现

### Phase 4: 输出 Context Packet

```markdown
# Context Packet: <标题>

## 目标
<1 句话——要什么结果，不是什么机制>

## 非目标
- <即使看起来好也不做的>

## 约束
- <前后必须保持为真的>

## 参考文件
- `path/to/file` — 当前要遵循的模式
- `path/to/test/` — 现有测试覆盖

## 验收标准
- [ ] 现有测试全部通过
- [ ] 新功能正常工作
- [ ] 性能指标达标

## 实施顺序
1. <第一步>
2. <第二步>

## 风险与回滚
- <可能怎么失败，怎么撤回>
```

## 注意事项
- 验收标准要可执行，不要"应该能工作"
- 非目标很重要，防止范围膨胀
- 规格先于实现，事后写的是文档不是规格
- 参考文件只列 3-10 个关键的，不要列太多
