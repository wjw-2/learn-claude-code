# 深入解析 Agent Harness 架构：从 0 到 1 构建企业级智能体系统

> **核心哲学：The Model IS the Agent, Code is the Harness.**
>
> 智能体（Agent）的本质是经过海量数据训练的神经网络模型（如 Qwen, GPT-4, Claude），它本身就具备感知、推理和行动的能力。我们的代码不是在“制造”智能，而是在打造一套“马具（Harness）”：为模型提供工具接口（感知）、执行环境（手脚）、记忆管理（大脑海马体）和安全护栏（底线）。
>
> 拒绝使用 LangChain、AutoGen 等过度封装的“玩具框架”，我们采用最纯粹的 Python，从零手搓一套接近工业级（Claude Code 级别）的底层架构。这不仅能让我们彻底掌控数据流，更是向面试官展示底层工程能力的最佳途径。

## 第一部分：架构演进与模块深度拆解 (S0 - S12)

### S0: 交互式循环 (REPL - Read-Eval-Print Loop)
**核心文件**: `main.py`

**架构解析**:
一切 Agent 的起点都是一个对话框。S0 阶段的目标是建立一个持久化的运行环境，使得大模型能够记住多轮对话的上下文。
1. **环境初始化**: 在程序启动时，通过 `dotenv` 加载 `.env` 文件中的 `DASHSCOPE_API_KEY`。这一步非常关键，它将敏感凭证与代码库解耦。
2. **全局状态维护**: 在 REPL 循环外部，我们定义了一个 `messages = []` 列表。这是整个 Agent 系统的“短期记忆”。
3. **事件循环**: 通过一个 `while True` 循环（Read），接收用户的标准输入（`input()`）。将用户输入封装为 `{"role": "user", "content": ...}` 追加入 `messages`，然后传递给核心的处理函数 `agent_loop`（Eval），最后将返回结果打印到终端（Print）。

**💡 面试深度拷问**:
- **Q: 为什么你的系统能记住多轮对话？API 不是无状态的吗？**
  - **A**: 没错，大模型的 REST API 本质上是无状态的（Stateless）。系统之所以有“记忆”，是因为我们在客户端（Harness 端）维护了 `messages` 列表。每次请求时，我们不仅发送当前的问题，而是将**过去发生的所有对话、工具调用记录、甚至错误日志**，作为完整的上下文（Context）一次性打包发给模型。记忆的负担由客户端的代码来承担。

### S1 & S2: 核心引擎与推理行动循环 (ReAct Loop & Tool Use)
**核心文件**: `loop.py`, `tool_use.py`

**架构解析**:
这是 Agent 能被称为“Agent”而不是“聊天机器人”的分水岭。
1. **ReAct 范式 (Reason + Act)**: 我们在 System Prompt 中要求模型遵循特定的思考方式。模型首先输出思考过程（Reasoning），然后生成特定的 JSON 结构来请求调用工具（Acting）。
2. **Function Calling (Tool Calls) 机制**: 
   - 在 `loop.py` 中，我们调用 Qwen 模型 API 时，不仅传入了 `messages`，还传入了 `tools=TOOLS_SCHEMA`（这是一个定义了所有可用工具名称、参数类型的 JSON Schema 列表）。
   - 模型返回的响应中，如果包含 `tool_calls` 字段，说明模型决定“动手”了。
3. **拦截与执行大循环**:
   - `loop.py` 内部有一个 `for step in range(15):` 的循环。
   - 提取模型请求的工具名称（`func_name`）和参数（`args`）。
   - 在 `tool_use.py` 的 `TOOL_HANDLERS` 字典中查找对应的 Python 函数（如 `bash`, `read_file`）并执行。
   - **关键步骤**：将执行结果组装成 `{"role": "tool", "tool_call_id": ..., "name": func_name, "content": result}` 的格式，**追加到 `messages` 中，并立即发起下一次 API 请求**。
   - 这个循环会自动持续，直到模型返回的响应中**没有 `tool_calls`**，此时视为任务收敛（Convergence），循环打破，返回最终的文本结果。

**💡 面试深度拷问**:
- **Q: 模型是如何执行本地命令的？它本身不是在云端吗？**
  - **A**: 这是很多人对 Agent 的误解。云端的模型绝对不具备执行本地代码的能力。模型仅仅是生成了一段符合约定的 JSON 字符串（比如 `{"name": "bash", "arguments": {"command": "ls"}}`）。真正的执行者是我们的 Python 脚本（Harness）。Python 拦截这段 JSON，解析出命令，调用底层的 `subprocess.run()` 在宿主机上执行，然后将终端的输出流（stdout/stderr）抓取下来，作为文本再喂给云端模型。模型是“大脑”，Harness 是“手脚”。
- **Q: 如果模型陷入了死循环，不停地调用工具报错怎么办？**
  - **A**: 这在复杂排错中极易发生。我们在 `loop.py` 中设置了 `max_steps = 15` 的硬性安全护栏（Hard Limit）。单次任务如果工具调用循环超过 15 次，系统会抛出异常强制阻断，防止陷入无限循环导致 API 账单爆炸。

### S3: 安全机制与沙箱隔离 (Security & Sandbox)
**核心文件**: `tool_use.py`

**架构解析**:
当 Agent 拥有了操作终端（`bash`）和文件系统（`read_file`, `write_file`）的权限后，它就变成了一个危险的实体。必须在 Harness 层建立坚固的护栏。
1. **路径沙箱 (Path Sandbox - `save_path` 函数)**:
   - 我们定义了 `WORKSPACE = Path(".").resolve()` 作为安全结界。
   - 任何涉及路径的工具调用，都会先经过 `save_path` 处理。它利用 Python 的 `path.is_relative_to(WORKSPACE)` 强制校验。如果模型试图通过 `../../` 读取宿主机的 `/etc/passwd` 或其他项目外的敏感文件，系统会直接拦截并抛出错误返回给模型，绝不执行。
2. **高危指令正则拦截与人工审批 (Human-in-the-loop)**:
   - 定义了 `DANGEROUS_COMMANDS_PATTERN`，使用正则表达式精确匹配 `del`, `rm`, `format`, `mkfs` 等高危终端命令，以及试图切换到系统根目录的操作（`cd /`）。
   - 在 `run_bash` 函数中，如果匹配到高危操作，主线程会立即阻塞（Pause）。终端会以醒目的红色字体打印警告，并要求人类用户输入 `y/n` 进行授权。这是“人类兜底”的核心防线。

**💡 面试深度拷问**:
- **Q: 你如何保证这个 Agent 在客户或同事机器上运行时的绝对安全？**
  - **A**: 依靠双重护栏：物理边界隔离和关键路径审批。物理边界通过代码级别的路径沙箱（Sandbox）实现，确保 Agent 的视野和触手被死死限制在当前工作目录下。关键路径审批通过正则拦截高危 Shell 命令，引入 Human-in-the-loop 机制，确保任何破坏性操作必须经过人类的显式授权。绝不把系统的生杀大权完全交给概率模型。

### S4: 三层记忆压缩架构 (Memory Management - 核心亮点)
**核心文件**: `context_compact.py`, `loop.py`

**架构解析**:
这是解决长文本任务导致 Token 溢出、费用高昂以及模型“注意力涣散（Lost in the middle）”的核心技术。我们没有使用简单的“滑动窗口丢弃”，而是设计了三层防御机制。
1. **第一层：微观压缩 (Micro Compact)**:
   - **触发时机**：每次工具执行完毕追加日志前。
   - **机制**：在长周期的 Debug 中，终端往往会打印成百上千行的堆栈信息。大模型只在紧接着的 1~2 步思考中需要这些信息。因此，`micro_compact` 函数会遍历 `messages`，**仅保留最近 3 次**工具调用的完整输出。对于更早之前的工具输出，如果其长度超过 1000 字符，则直接将其内容替换为占位符 `[Previous tool output compacted]`。
   - **效果**：以极低的计算成本，悄无声息地剪除了历史包袱，同时完美保全了当前的思考连续性和原始的 System Prompt。
2. **第二层：自动宏观压缩 (Auto Compact)**:
   - **触发时机**：当 `estimate_tokens` 估算出上下文 Token 达到危险阈值时。
   - **机制**：触发“掀桌子”式重建。系统暂停当前循环，将冗长的 `messages` 发送给另一个模型实例（Summarizer），并提示其：“你是会话压缩器。请保留目标、已完成操作、未完成事项、关键路径和错误。” 拿到精炼的摘要后，**清空旧的 `messages`**，用仅包含系统提示、压缩摘要和伪造确认的 3 条消息重建上下文。
3. **第三层：主动请求压缩 (Manual Compact)**:
   - **触发时机**：模型自我感知混乱时。
   - **机制**：向模型提供一个 `compact_context` 工具。当模型在复杂任务中觉得当前上下文过于杂乱时，可以主动调用此工具，Harness 收到信号后会立即触发第二层的自动宏观压缩。

**💡 面试深度拷问**:
- **Q: 如果上下文太长，你们是怎么处理的？直接把最前面的聊天记录删掉吗？**
  - **A**: 绝对不能直接删掉最前面的记录。因为最前面往往包含了系统设定（System Prompt）和用户最初的全局任务目标。如果直接丢弃，Agent 会瞬间“失忆”，不知道自己是谁、要干嘛。
  - 我们的方案是三层记忆架构……（详细描述微观剔除长日志和宏观模型总结重建机制）。这种方法既保全了大局观，保留了之前踩坑的错误记忆，又释放了巨大的 Token 空间。

### S5: 状态机防迷失与任务催办 (Task Planning & Nagging)
**核心文件**: `todo_write.py`, `loop.py`

**架构解析**:
在处理复杂多步任务（如搭建一个带有前后端和数据库的全栈项目）时，大模型极易陷入局部细节（比如死磕一个 CSS 样式），从而忘记全局进度。
1. **结构化任务管理 (`todo` 工具)**:
   - 我们提供了一个 `todo` 工具，强制要求模型在接手复杂任务时，先调用该工具列出分解的子任务（支持 `pending`, `in_progress`, `completed` 状态）。
2. **追踪与催办机制 (Nagging Mechanism)**:
   - 在 `loop.py` 的主循环中，维护了一个计数器 `rounds_since_todo`。每次循环加 1，当模型调用 `todo` 工具更新进度时清零。
   - 如果大模型像个“埋头苦干却不看路”的员工，连续 3 轮（`rounds_since_todo >= 3`）都在查代码、改代码、报错，而没有去更新全局的 Todo 列表，系统会**强行向 `messages` 中注入一条指令：`<reminder>Update your todos.</reminder>`**。
   - **效果**：模型看到这条提醒，会被强制拉出局部的思维管线，退回全局视角审视当前进度，从而及时调整错误的方向。

### S6 & S10: 多 Agent 协作与上下文隔离 (Sub-Agents & Teams)
**核心文件**: `subagent.py`, `tool_use.py`, `agent_teams.py`

**架构解析**:
“多 Agent 协作”在很多框架里被包装得玄之又玄，但在这个底层 Harness 中，我们揭示了它的工程本质。
1. **委派机制 (`delegate_task` 工具)**:
   - 主 Agent 可以通过调用 `delegate_task`，将某个复杂的子任务（比如“帮我写一个正则表达式来匹配邮箱”）外包出去。
2. **底层实现 (Context Isolation 上下文隔离)**:
   - 当 `delegate_task` 被触发时，`subagent.py` 会被唤醒。它会初始化一个**全新且空白的 `messages` 列表**，并以收到的 prompt 作为起点，开启一个完全独立的新一轮 ReAct 循环。
   - 子 Agent 完成任务后，将其最终的文本结果作为字符串返回。主 Agent 接收到这个字符串，将其作为工具的返回值写入自己的 `messages` 中，继续往下执行。
3. **团队协议 (Team Protocols)**:
   - 进阶的协作允许注册特定的角色（前端专家、数据库专家），并通过 `team_assign` 和 `team_inbox` 模拟现实工作中的异步消息传递和任务分发。

**💡 面试深度拷问**:
- **Q: 你们系统里多 Agent 是怎么通讯的？引入多 Agent 的根本目的是什么？是为了让系统显得更智能吗？**
  - **A**: 并不是为了显得智能，多 Agent 的根本工程目的是**实现“上下文隔离（Context Isolation）”**。
  - 想象一下，主 Agent 正在统筹一个大项目，中间遇到了一个极度棘手的环境配置问题。如果主 Agent 自己去排查，几十次的终端报错日志会瞬间塞满它的上下文，把原本清晰的项目规划冲刷得一干二净。
  - 通过 `delegate_task`，主 Agent 开启一个子 Agent 去解决这个脏活累活。子 Agent 在自己独立的上下文中疯狂试错，无论产生多少垃圾日志，都不会污染主 Agent 的大脑。最后子 Agent 只需要向主 Agent 汇报一句“配置已解决，路径在 xxx”即可。这才是多 Agent 架构真正的威力所在。通讯本质上就是函数调用和返回值（I/O 传递）。

### S7 & S8: 并发执行、后台任务与 DAG 编排 (Concurrency & DAG)
**核心文件**: `background_task.py`, `dag_task_system.py`

**架构解析**:
让 Agent 从单线程的“串行工人”进化为能够统筹多线程的“包工头”。
1. **后台任务 (`run_background` 工具)**:
   - 遇到 `npm install` 或 `pytest` 这种动辄耗时几分钟的命令，如果让主线程死等，API 会超时，而且浪费算力。
   - 该工具利用 Python 的 `threading.Thread` 创建独立线程执行命令。主 Agent 触发后立刻收到“已在后台运行”的通知，可以继续去修改其他文件。后台任务自带 `threading.Lock` 保证多线程写入日志时的状态安全。
2. **有向无环图 (DAG Task System)**:
   - 面对超级复杂的项目，提供 `dag_create` 和 `dag_update` 工具。
   - 允许定义任务依赖（例如：任务 B 的 `blocked_by` 包含任务 A 的 ID）。
   - 调度引擎会自动检查依赖树，只有当任务 A 被标记为 `completed` 时，任务 B 才会解除阻塞（Unblocked），被投递进执行队列。这使得 Agent 具备了处理复杂拓扑工程项目的能力。

## 第二部分：面试实战策略（应对高频/刁钻提问）

在面试中，仅仅描述功能是不够的。你需要向面试官展示你对底层原理的透彻理解，以及在实际工程中做出的**Trade-off（权衡取舍）**。

### 🚨 高频难题 1："为什么你们团队要从零手写 Agent 框架，而不是直接使用 LangChain 或 AutoGen？这是在重复造轮子吗？"
> **🌟 高分回答策略：展现对主流框架弊端的深刻洞察，以及对“控制力”的追求。**
> "并不是重复造轮子。LangChain 确实降低了入门门槛，但它在处理企业级复杂 Agent 时往往显得过于臃肿（Over-engineered）。
> 它的各类 Chain 和黑盒 Prompt 封装（所谓 Prompt Plumbing），本质上是在掩盖模型本身的智能，试图用死板的代码逻辑去硬编码行为。而且它剥夺了开发者对底层 ReAct 循环的精细控制权。
> 在我们的实际业务中，我们需要极高精度的上下文控制（比如我们自研的三层记忆压缩机制，LangChain 很难优雅地插入这种微观控制）、定制化的安全拦截（底层的路径沙箱机制）以及细粒度的状态机防迷失策略（Todo Nag）。使用原生的 Python 直接对接大模型 API（如 Qwen/OpenAI 兼容接口），能让我们像做外科手术一样精准控制数据流向，系统更稳定、调试更透明，且避免了沉重的框架依赖开销。我们认为：The Model IS the Agent, Code is just the Harness。"

### 🚨 高频难题 2："如果在执行复杂排错任务时，模型陷入了『修改 Bug A -> 引入 Bug B -> 撤销 -> 再次引入 Bug A』的死循环，你的架构如何解决？"
> **🌟 高分回答策略：展示立体的容错设计和工程兜底能力。**
> "这在 Agent 辅助编程中非常常见，也就是所谓的『逻辑打墙』。我们的架构不指望模型自己能 100% 绕出来，而是通过三个层面进行干预：
> 1. **物理阻断（Hard Limit）**：在 `loop.py` 设定 `max_steps`（通常为 15 步），如果循环超过这个阈值，Harness 会强行终止循环并抛出异常，防止无效的 API 消耗。
> 2. **认知重置（Sub-Agent 隔离）**：当遇到复杂环境排错时，我们会引导主 Agent 调用 `delegate_task` 委派给子 Agent。子 Agent 在一个干净的上下文中去尝试修复。即使子 Agent 陷入死循环崩溃，主 Agent 依然能保持清醒，并可以选择更换方案。
> 3. **宏观拉扯（Todo Nag）**：通过 `rounds_since_todo` 机制，强制模型在陷入局部细节太久时，强行插入 `<reminder>Update your todos.</reminder>`，这相当于在它耳边大喊一声『看看全局计划！』。这往往能打破模型的局部最优解，让它意识到当前的死磕方向是错的，从而主动切换思路。"

### 🚨 高频难题 3："你在做这个底层 Harness 的过程中，遇到的最大技术挑战是什么？具体是怎么解决的？"
> **🌟 高分回答策略：抛出『上下文膨胀（Context Bloat）』这个痛点，展现你的深度思考方案。**
> "最大的挑战是如何在有限且昂贵的 Token 窗口内，让 Agent 稳定地完成长达几十步、跨越多个文件的代码编写任务。
> 早期版本中，只要任务稍微长一点，执行几次带有长日志输出的 Shell 命令，系统跑着跑着就会报 'Token Exceed' 错误，或者模型因为输入太长开始胡言乱语（注意力衰减）。
> 解决方案是我设计并实现了**三层记忆压缩架构**。特别是其中的『微观压缩』层让我最满意：我通过代码识别历史 `messages` 中超过 1000 字符的非近期工具输出，将其替换为 `[Previous tool output compacted]` 的占位符。这个极其轻量级的算法，在极低的代码成本下，削减了超过 60% 的无用日志消耗，完美保全了模型执行任务的连贯性，让系统能够支撑无限长度的复杂任务推演。"

### 🚨 高频难题 4："这个项目里用到了哪些经典的设计模式？"
> **🌟 高分回答策略：技术术语的准确运用，展现良好的代码品味。**
> "为了保证框架的可扩展性，我大量使用了面向对象和设计模式：
> 1. **策略模式 (Strategy Pattern)**: 在 `tool_use.py` 的 `TOOL_HANDLERS` 字典中，通过统一的键值映射，将工具名称（如 'bash', 'edit_file'）分发到不同的处理函数。这彻底解耦了工具的定义与核心的 ReAct 循环，新增工具只需要注册一个 Lambda 表达式即可，完全符合开闭原则（OCP）。
> 2. **状态模式 (State Pattern)**: `autonomous_agent.py` 中的 `AUTO_STATE` 控制了 Agent 的空闲/执行状态机。通过显式的状态对象管理行为切换。
> 3. **单例模式 (Singleton Pattern)**: 像 `BG_TASK_MANAGER`（后台任务管理器）这种需要全局共享资源和线程池的对象，被设计为单例，确保全生命周期内任务队列的唯一性。
> 4. **沙箱模式 (Sandbox)**: `save_path` 实现了严格的目录隔离边界，这是一种经典的安全设计模式。"

### 🚨 高频难题 5："Agent 究竟和传统的 RPA（机器人流程自动化）有什么本质区别？"
> **🌟 高分回答策略：一针见血地指出『决策权』的归属。**
> "传统的 RPA（比如各类拖拽式的自动化平台）是『基于规则的（Rule-based）』。开发者需要预先画好所有的流程图（If A happens, do B）。系统的上限就是开发者脑海中预设的逻辑树上限。
> 而我们构建的 Agent 架构是『基于意图的（Intent-based）』。我们只给模型提供目标（System Prompt）和一组原子工具（Harness），至于遇到问题时是先读文件、还是先运行终端排错，全由模型内部的神经网络在运行时刻自主推理（ReAct）决定。Agent 具备应对未知环境和未预设错误的泛化能力，这是 RPA 绝对无法做到的。"

## 第三部分：代码级全链路拆解（从用户输入到任务完成）

这一部分按“真实执行顺序”讲清楚系统在一轮任务中的每一步。面试官最喜欢问“你的系统跑起来到底发生了什么”，你只要按这个顺序讲，逻辑就非常完整。

### 0）冷启动阶段（进程刚启动）
1. `main.py` 先加载环境变量，校验 `DASHSCOPE_API_KEY` 是否存在。
2. 校验失败直接 `sys.exit(1)`，避免把错误拖到运行期。
3. 初始化 `messages` 与 `SYSTEM`，打印欢迎语，进入 REPL 循环。

### 1）接收任务阶段（用户输入）
1. 用户输入被包装成：
   - `{"role": "user", "content": query}`
2. 这一条被追加到 `messages`，作为本轮推理的最新目标。
3. 进入 `agent_loop(query, messages)`。

### 2）循环前预处理阶段（压缩/状态注入）
1. `micro_compact(messages)` 执行第一层压缩：
   - 只压缩历史里“较早且较长”的 tool 输出；
   - 最近 N 次工具结果保持完整，保证短期可推理性。
2. `AUTO_STATE.tick` 机制在每轮会更新行为状态（是否调用工具、是否进入空转）。
3. `BG_TASK_MANAGER.collect_finished()` 会把后台任务结果注入到 `messages`。

### 3）大模型调用阶段（Function Calling）
1. 发起 `client.chat.completions.create(...)`：
   - `model=MODEL`
   - `messages=messages`
   - `tools=TOOLS_SCHEMA`
2. 返回包中读取 `response.choices[0].message`。
3. 使用 `message.model_dump(exclude_none=True)` 转成纯 dict，塞回 `messages`。

### 4）收敛判断阶段（是否还需要动作）
1. 若 `not message.tool_calls`：
   - 认为任务“已收敛”
   - 返回 `message.content` 到前台
2. 若存在 `tool_calls`：
   - 进入工具执行分支
   - 逐个解析 `function.name` 与 `function.arguments`

### 5）工具执行阶段（Harness 执行真实动作）
1. 在 `TOOL_HANDLERS` 中查找 handler。
2. 找到则执行；找不到返回 `未知工具` 错误给模型。
3. 每个工具输出都以 `role=tool` 格式写回 `messages`：
   - `tool_call_id`
   - `name`
   - `content`
4. 模型据此继续下一轮“Reason -> Act”。

### 6）防迷失与记忆治理阶段
1. `rounds_since_todo += 1`：统计模型多久没更新任务计划。
2. 连续多轮不更新就注入 `<reminder>Update your todos.</reminder>`。
3. 若触发 `manual_compact_requested`，立即执行 `auto_compact`。
4. 如果循环超过 `max_steps`，硬终止，避免无限 token 消耗。

## 第四部分：S0-S12 的“面试版本”精讲（每个阶段讲清目标、实现、价值、边界）

下面每个阶段都包含四个子维度：  
**要解决什么问题 -> 代码怎么做 -> 为什么这么做 -> 局限在哪里。**

### S0：REPL 与最小可运行 Agent

**要解决的问题**
- 如何让“聊天模型”变成“可持续执行任务”的系统入口。

**代码怎么做**
- 通过 `while True` 接受用户输入。
- 维护全局 `messages` 列表作为客户端记忆。
- 所有请求都走统一 `agent_loop`。

**为什么这么做**
- API 无状态，客户端必须承担状态。
- 这是最小闭环，先跑通再迭代复杂能力。

**局限与边界**
- 初期没有权限隔离、没有任务图、没有并发能力。
- 一旦任务复杂，马上会暴露上下文膨胀问题。

**面试一句话总结**
- “S0 的核心是把无状态 API 封装成有状态会话系统，这是 Agent 的第一个门槛。”

---

### S1：终端工具执行（bash 工具）

**要解决的问题**
- 模型只会“说”，不会“做”；必须给它可执行动作。

**代码怎么做**
- 注册 `bash` 工具。
- `run_bash` 使用 `subprocess.run` 执行命令。
- 设置超时避免死锁。

**为什么这么做**
- 工程任务大量依赖 shell 命令（安装、测试、构建、查询）。
- 没有 bash 能力，Coding Agent 基本不可用。

**局限与边界**
- 命令输出可能非常长，导致上下文污染。
- 没有审批会有安全风险。

**面试一句话总结**
- “S1 把模型从‘文本生成器’升级成‘能操作真实环境的执行体’。”

---

### S2：统一工具注册中心（Schema + Handler）

**要解决的问题**
- 工具越来越多后，如何统一治理和扩展。

**代码怎么做**
- `TOOLS_SCHEMA`：告诉模型“你可以调用什么”。
- `TOOL_HANDLERS`：告诉系统“工具被调用后由谁执行”。
- 名称匹配 + 参数解析 + 统一回写 `role=tool`。

**为什么这么做**
- 明确“声明层”（给模型看）与“执行层”（给 Python 跑）。
- 新增工具只需增量注册，开闭原则友好。

**局限与边界**
- 工具名称冲突要治理。
- schema 与 handler 参数不一致会导致调用失败。

**面试一句话总结**
- “S2 的本质是把工具系统从散乱函数升级为可维护的插件总线。”

---

### S3：安全护栏（路径沙箱 + 危险命令审批）

**要解决的问题**
- Agent 有写文件/跑命令能力后，必须防止误操作。

**代码怎么做**
- `save_path` 强制路径必须在 `WORKSPACE` 内。
- `_requires_human_approval` 识别危险命令并触发 y/n 审批。

**为什么这么做**
- 真实生产环境里“安全先于智能”。
- 工具能力越强，安全边界越必须硬编码。

**局限与边界**
- 正则黑名单不是完美防御，需持续迭代。
- 审批依赖人工在线，自动化流水线里要设计替代策略。

**面试一句话总结**
- “S3 的价值是把‘默认信任模型’改成‘默认不信任模型’。”

---

### S4：任务管理（Todo + Nagging）

**要解决的问题**
- 模型容易陷入局部修修补补，忘记全局目标。

**代码怎么做**
- `todo` 工具维护任务列表和状态。
- `rounds_since_todo` 追踪连续未更新轮次。
- 超阈值注入 reminder，强制模型回到计划层。

**为什么这么做**
- 大模型擅长局部推理，不擅长长期项目管理。
- 外置任务状态是“系统级认知补丁”。

**局限与边界**
- todo 文本质量依赖模型本身。
- 如果任务拆分不合理，nag 也只能提醒，不能替代规划能力。

**面试一句话总结**
- “S4 是给模型装一个 PM（项目经理）。”

---

### S5：Sub-Agent（委派与上下文隔离）

**要解决的问题**
- 主任务中经常有“脏活累活”会污染主上下文。

**代码怎么做**
- 主 Agent 调用 `delegate_task`。
- 子 Agent 在独立 `messages` 中运行自己的小循环。
- 子任务结果以摘要字符串回传主 Agent。

**为什么这么做**
- 不是为了“并行炫技”，而是为了隔离噪音，保持主线稳定。

**局限与边界**
- 子任务质量依然依赖模型质量。
- 子 Agent 结果如果太粗糙，主 Agent 还要二次修正。

**面试一句话总结**
- “S5 的工程价值是 context isolation，不是盲目多智能体。”

---

### S6：Skills（可复用能力注入）

**要解决的问题**
- 常见任务需要稳定模板和统一方法论。

**代码怎么做**
- 扫描 `.skills/**/SKILL.md`。
- 解析 Frontmatter 与正文，建立索引。
- `load_skill(name)` 按需注入技能内容。

**为什么这么做**
- 把经验沉淀成可调用资产，减少重复提示词工程。

**局限与边界**
- 技能文件质量决定效果上限。
- 缺少版本治理会导致“技能漂移”。

**面试一句话总结**
- “S6 是把团队经验产品化，而不是每轮都从零提示。”

---

### S7：后台任务（异步执行）

**要解决的问题**
- 长命令阻塞主循环，交互体验差。

**代码怎么做**
- `run_background` 开线程执行命令。
- 使用锁保护任务状态。
- 任务完成后由主循环收集并注入消息。

**为什么这么做**
- 把“等待时间”变成“并行工作时间”。

**局限与边界**
- 线程过多会争抢资源。
- 输出截断后可能丢细节，需要二次读取日志。

**面试一句话总结**
- “S7 让 Agent 从串行执行器升级为可调度系统。”

### S8：DAG 任务系统（依赖编排）

**要解决的问题**
- 复杂任务存在前后依赖，不能靠线性 todo 表达。

**代码怎么做**
- `dag_create` 建任务节点与 `blocked_by` 依赖。
- `dag_update(completed)` 后自动解锁后继任务。
- JSON 持久化，保证重启可恢复。

**为什么这么做**
- 让 Agent 具备处理“拓扑结构任务”的能力。

**局限与边界**
- 当前是简化 DAG，不含优先级/资源约束/失败重试图策略。

**面试一句话总结**
- “S8 的核心是从‘计划列表’升级到‘计划图’。”

### S9：Team（成员注册与消息收件箱）

**要解决的问题**
- 如何在系统层表达“角色分工”。

**代码怎么做**
- `team_register` 建成员。
- `team_assign` 发送任务到成员 inbox。
- `team_inbox` 拉取并消费消息。

**为什么这么做**
- 用角色抽象提升解释性，方便面试讲“组织协作”。

**局限与边界**
- 当前是轻量机制，不是完整多进程调度平台。

**面试一句话总结**
- “S9 是把单 Agent 思维升级到组织协作抽象。”

### S10：Protocol（审批协议）

**要解决的问题**
- 敏感动作需要可审计决策链。

**代码怎么做**
- `protocol_request` 创建请求。
- `protocol_decide` 进行 approve/reject。
- `protocol_get` 查询状态。

**为什么这么做**
- 治理能力是企业场景刚需。

**局限与边界**

**面试一句话总结**
- “S10 是把‘口头审批’落地成结构化协议。”

### S11：Autonomous Mode（自治状态机）

**要解决的问题**
- 希望 Agent 在合适场景自动推进。

**代码怎么做**
- `AUTO_STATE.enable/disable/tick` 管理自治状态和空闲轮次。

**为什么这么做**

## 第五部分：关键设计模式与工程取舍（可直接背诵）

### 1）策略模式（Strategy）
- 体现位置：`TOOL_HANDLERS`
- 价值：新增工具不用改主循环，只做注册。
- 取舍：牺牲了一点静态类型提示，换来极高可扩展性。

### 2）状态模式（State）
- 体现位置：`AUTO_STATE`
- 价值：把“是否自治、空闲计数、动作反馈”统一成状态机。
- 取舍：状态太少时看起来“重”，但为后续扩展留好接口。

### 3）门面模式（Facade）
- 体现位置：`agent_loop` 作为总入口。
- 价值：上层只关心输入和输出，不关心内部压缩、工具、后台线程细节。
- 取舍：门面太厚会变“上帝函数”，需要持续重构边界。

### 4）沙箱模式（Sandbox）
- 体现位置：`save_path` + 危险命令审批
- 价值：把模型行动范围限制在可控边界。
- 取舍：安全越严格，自动化自由度越低，这是必然权衡。

### 5）生产者-消费者思想（简化）
- 体现位置：后台任务线程产生结果，主循环消费结果。
- 价值：避免阻塞，提升吞吐。
- 取舍：需要状态同步和输出截断策略。

## 第六部分：面试官最爱追问的“故障场景”与标准回答

### 场景 A：模型持续调用不存在的工具
**问题本质**
- schema 和 prompt 认知不一致，或模型幻觉。

**系统反应**
- `handler` 查不到，返回 `未知工具`。
- 错误作为 `tool` 消息回灌，模型下一轮可自我修正。

**你该怎么答**
- “我把错误也结构化写回上下文，让模型具备自愈机会；并用 max_steps 防止错误自愈失败时无限循环。”

### 场景 B：终端命令卡死
**问题本质**
- 子进程阻塞。

**系统反应**
- 前台命令 `timeout=60`；
- 后台命令 `timeout=300`；
- 超时后返回失败并进入下一轮决策。

**你该怎么答**
- “超时是硬性工程护栏，宁可失败也不让系统被单命令拖死。”

### 场景 C：上下文超长导致模型失真
**问题本质**
- Token 超限与注意力衰减。

**系统反应**
- 微观压缩先清理冗余日志；
- 达阈值后自动摘要重建；
- 模型也可手动请求 compact。

**你该怎么答**
- “我们不是粗暴截断，而是分层压缩，保留任务目标与关键错误轨迹。”

### 场景 D：模型误删文件风险
**问题本质**
- 工具能力过强且不可控。

**系统反应**
- 路径沙箱拦截越界；
- 高危命令触发人工审批；
- 用户拒绝立即返回失败。

**你该怎么答**
- “默认不信任模型，所有破坏性路径都需要硬编码防线与人类授权。”

## 第七部分：性能、成本与稳定性指标（面试可量化表达）

你可以用下面这套口径把“我做了很多功能”变成“我做了工程优化”。

### 指标 1：单任务步骤上限
- 指标定义：单次任务允许的最大 ReAct 步数。
- 当前策略：主循环 15 步，子循环 10 步。
- 工程价值：防止无限循环造成成本失控。

### 指标 2：上下文压缩命中率
- 指标定义：每 N 轮任务中触发微观压缩/宏观压缩的比例。
- 工程价值：衡量记忆治理是否有效。

### 指标 3：后台任务平均等待收益
- 指标定义：后台任务执行期间主循环完成的额外有效动作数。
- 工程价值：衡量并发带来的真实吞吐提升。

### 指标 4：危险命令拦截数
- 指标定义：触发审批的命令数量与拒绝比例。
- 工程价值：衡量安全机制是否真正发生作用。

### 指标 5：任务收敛率
- 指标定义：在 `max_steps` 内完成任务的比例。
- 工程价值：衡量整体策略质量与 prompt/tool 设计质量。

## 第八部分：你可以主动说出的“当前不足与升级路线”（这会非常加分）

### 已识别不足 1：自治工具参数口径不一致
- 现象：schema 使用 `action`，handler 读取 `enabled`。
- 风险：模型正确调用 schema 仍可能触发行为偏差。
- 升级思路：统一为 `action`，在 handler 做枚举映射。

### 已识别不足 2：协议系统缺少角色权限模型
- 现象：当前 protocol 是状态流，不是 RBAC。
- 风险：审批主体可信度表达不足。
- 升级思路：增加角色、权限、审计日志和签名字段。

### 已识别不足 3：DAG 缺少失败重试策略
- 现象：任务失败后缺少自动重试与熔断策略。
- 升级思路：增加 retry/backoff、失败分支和告警机制。

### 已识别不足 4：可观测性仍偏日志级
- 现象：缺统一 trace_id、指标面板。
- 升级思路：接入结构化日志与指标系统，追踪每轮 token 与工具时延。

### 并发与调度类

23. **问：DAG 相比 todo 的增量价值？**
