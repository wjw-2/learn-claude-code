# Debug

找到根因，修复它，证明它有效。

## 铁律

**没有根因调查，不准提修复方案。**

## 四阶段协议

### Phase 1: 根因调查
1. 仔细阅读错误信息和完整堆栈
2. 稳定复现问题（精确步骤）
3. 检查最近变更：`git diff`, `git log --oneline -10`, 新依赖
4. 多组件系统：在组件边界打日志，定位失败层
5. 追踪数据流：坏值从哪里来？逆向追踪到源头

### Phase 2: 模式分析
1. 找同代码库中的正常工作示例
2. 完整阅读参考实现，不要扫读
3. 列出所有差异（无论多小）
4. 理解依赖关系：配置、环境、假设

### Phase 3: 假设与测试
科学方法，一次一个实验：
1. 写下单个假设："我认为 X 导致 Y，因为 Z"
2. 设计实验：什么能证明/证伪？最小变更，单变量
3. 运行实验，观察结果
4. 证伪→消除，重新假设；支持→进一步验证；模糊→缩小范围重试

### Phase 4: 实现
1. 先写失败测试（复现 bug）
2. 确认测试因正确原因失败
3. 单次修复（针对根因）
4. 验证：测试通过，无回归，问题解决

## 调试优先级

| 类型 | 信号 | 方法 |
|------|------|------|
| 配置问题 | 环境变量缺失/格式错误 | 先检查配置，再读代码 |
| 运行时错误 | 异常、崩溃 | 堆栈→源码→状态 |
| 类型错误 | TS 报错 | 读错误，检查类型定义 |
| 构建失败 | 打包器报错 | 检查依赖、配置 |
| 行为不匹配 | "做了 Y，应该做 X" | 追踪代码路径 |

## 红线（停止并回到 Phase 1）
- "先快速修复，之后再调查"
- "改 X 试试看"
- 同时做多个变更
- 追踪数据流之前提方案
- 每个修复都揭示出新问题

## 输出格式
- **根因**: 实际出了什么问题
- **修复**: 如何解决的
- **验证**: 可观测证据证明修复有效
