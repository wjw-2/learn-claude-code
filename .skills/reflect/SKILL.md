# Reflect

会话结束时进行结构化回顾，提取可复用的知识。

## 工作流

### 1. 收集证据
从会话中找：
- **用户纠正**: 用户说"不对"、"应该是"的地方——这是系统失误
- **可用但未使用的信息**: 文件或配置存在但 agent 没找到
- **多次尝试**: 需要 >1 次才成功的地方——缺少防护措施
- **等待时间**: 用户等了但没产出价值的地方

### 2. 按层级修复

| 层级 | 检查什么 | 修复方式 |
|------|---------|---------|
| 指令 (CLAUDE.md) | 缺少指导、过时规则 | 更新指令 |
| Skills | 缺少 skill、描述不对 | 创建/修复 skill |
| Hooks | 缺少前置检查 | 添加 hook |
| 工具/环境 | 缺少 MCP、工具损坏 | 修复工具 |
| Agent 推理 | 有正确上下文但做错决定 | 最低优先级 |

### 3. 分类和编码

**工作流发现 > 代码发现**。一个缺失的"如何运行 QA"比一个 grep 脚本价值更高。

编码层级（从高到低）：
类型系统 > Lint 规则 > Hook > 测试 > CI > Skill > 文档

### 4. 输出

```
## 工作流失败
- [纠正/死胡同] → [系统缺口] → [修复]

## 代码级发现
- [模式] → [编码目标] → [修复]

## 未编码
- [发现]: [明确的不编码理由]
```

## 注意事项
- 回顾不产生提交就是浪费时间
- 在错误层级编码：用 hook 能解决的不要写文档
- 过时信息比没有信息更糟——不确定就删除
